# Implementation Plan
**Project:** BookItApp — Room Booking
**Date:** 2026-03-17
**Source:** BRD `docs/requirements/BRD.md` + Issues in `issues/`

## Overview
BookItApp delivers meeting-room booking to authenticated users. The feature is decomposed across 6 Issues: 1 DATABASE, 2 BACKEND, 2 FRONTEND, and 1 PLAYWRIGHT. The work is organised into two sprints. Sprint 1 delivers the full primary slice — room browsing, booking creation, and dashboard — so the app is usable end-to-end after a single sprint. Sprint 2 adds the Booking Cancellation extension and runs the shared Playwright test suite. Human roles involved are: Architect (DATABASE), Backend Dev (BACKEND + unit tests), UI Dev (FRONTEND), and QA Engineer (PLAYWRIGHT).

## Risk Register

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| R-001 | BACKEND issue merged before DATABASE migration runs — foreign key errors at runtime | High | Enforce Assignment Order: Step 2 (#5) must not be assigned until Step 1 (#6) PR is merged and `npm run db:migrate` has completed successfully |
| R-002 | Coding agent modifies pre-built files (auth middleware, App.tsx, index.ts) | High | Review every PR diff before merging; if any pre-built file appears in the changeset, reject and re-assign the issue with an explicit "do not touch" note |
| R-003 | Seed data missing or empty — frontend shows blank page; Playwright tests find no records | High | Verify seed output after `npm run db:seed`: must contain ≥ 4 Room rows and ≥ 3 ACTIVE Booking rows for test@example.com before marking DATABASE issue as done |
| R-004 | `npx prisma` invoked instead of `npm run db:*` — picks up global Prisma v7+ and fails | High | AGENTS.md explicitly prohibits `npx prisma`; check the DATABASE PR's run commands and reject if `npx prisma` appears |
| R-005 | BookingStatus defined as `String` instead of a Prisma enum — overlap logic and status filtering break | Medium | Review schema.prisma in the DATABASE PR: `BookingStatus` must be declared as a Prisma `enum`, not a String field with a comment |
| R-006 | Missing `data-testid` attributes on interactive elements — Playwright selectors fail | Medium | Cross-reference the data-testid table in Issue #8 and Issue #7 against the rendered HTML before approving FRONTEND PRs; run Playwright in headed mode locally to confirm all selectors resolve |
| R-007 | Overlap detection implemented only in the UI, not enforced at the API layer (violates NFR-004) | Medium | Confirm that the POST /api/bookings controller queries for existing ACTIVE bookings for the same room/date/time window before inserting; verify with a direct API test (curl or Jest) |
| R-008 | `GET /api/bookings/my` route shadowed by `/:id` — Express routes "my" as a booking ID | Medium | BACKEND Issue #5 explicitly requires registering `/my` before `/:id`; verify route order in the bookings router file during PR review |
| R-009 | FRONTEND build fails with TypeScript strict-mode errors — blocks Playwright tests | Low | Run `npm run build` (from `src/frontend/`) after the FRONTEND PR is opened; CI must pass before merging |

## Success Criteria

The feature is complete when:
- [ ] All 6 Issues are merged in Assignment Order (Steps 1 → 6)
- [ ] `npm run build` succeeds with zero TypeScript errors (both backend and frontend)
- [ ] All Playwright E2E tests pass (`npx playwright test`)
- [ ] The HomePage shows the room list, booking form, and booking dashboard with real data — not "Features coming soon"
- [ ] Every BRD functional requirement (FR-001 through FR-006) has a corresponding acceptance criterion verified by a passing test
